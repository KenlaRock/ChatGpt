# Collaboration Provenance

## Purpose

NullForge may use human contributors, AI collaborators, model candidates, automated tools and generative platform features during research and development. This document defines how those contributions are attributed without treating the source itself as scientific authority.

A useful idea remains eligible regardless of origin. Promotion depends on verification, reproducibility and fit with the project rules.

## Stable actor identity

Contributor identity and runtime identity are separate fields.

- `actor_id` is the stable project identity used across contributions.
- `actor_type` describes whether the contributor is human, AI collaborator, model under test, automated tool or external reviewer.
- `platform` records the service or execution environment.
- `model_or_runtime` records the specific model, version or toolchain used for that contribution.
- `role` records the function performed, such as owner, implementation, review, QA, research support or test candidate.

A model or runtime change must not silently create a new project identity, and a stable project identity must not imply that every later runtime has identical behavior.

## Recommended contribution record

```yaml
actor_id: NF-ACT-EXAMPLE
actor_type: ai_collaborator
platform: example-platform
model_or_runtime: example-runtime-version
role:
  - review
  - implementation_support
task_id: EXAMPLE_TASK_001
date: 2026-07-12
source_feature: direct_assistant_session
reliability: exploratory
evidence_class: REFERENCE_NOTE
decision: pending
approved_by: human_project_arbiter
```

Only fields appropriate for the public repository may be published. Internal workspace identifiers, private URLs, access details and raw conversations remain prohibited by `PUBLICATION_BOUNDARY.md`.

## Source classes

### Direct project contribution

Work produced during an explicitly scoped project task. It still requires the normal proof, review and test procedures.

### Authored collaborator review

A retained review artifact with an identifiable reviewer, declared scope, date, limitations and freshness status. Review findings may be accepted, modified, deferred or rejected.

### Model candidate or multimodal test source

Output used to compare model behavior, test multimodal interpretation or challenge an existing result. Candidate output is not evidence merely because it was generated by a capable model.

### Generative secondary commentary

Generated summaries, discussions or audio-overview features are treated as exploratory secondary commentary.

For NotebookLM Audio Overview, record:

```yaml
platform_address: notebooklm.google.com
feature: Audio Overview
source_class: generative_secondary_commentary
reliability: exploratory
caveat: non-authoritative; independently verify before adoption
```

The project should retain the adopted point and its decision trail rather than archiving a full generated transcript unless the transcript itself has a specific reviewed research purpose.

## Decision states

Each retained proposal or finding should be marked as one of:

- `PENDING`
- `ADOPTED`
- `ADOPTED_WITH_MODIFICATION`
- `DEFERRED`
- `REJECTED`
- `SUPERSEDED`

Attribution records where an idea came from. Decision status records what the project did with it. Neither field substitutes for evidence classification.

## Promotion boundary

Collaboration and feedback records are working context, not canonical scientific truth.

Before an internal contribution is promoted to the public repository:

1. identify the source class and contribution role;
2. remove private links, identifiers and raw workspace material;
3. classify claims as policy, hypothesis, measured evidence or another allowed evidence class;
4. preserve relevant provenance and failed-attempt context;
5. run the public-boundary check;
6. review the resulting diff through the normal pull-request procedure.

## Non-equivalences

- contributor attribution is not scientific validation;
- model output is not measured fact;
- feedback source is not authority;
- collaboration records are not release state;
- a generated summary is not a substitute for its underlying sources;
- public documentation must not link back into private collaboration workspaces.
